1) Účel a smysl systému
1.1 Cíl produktu

Vybudovat jednotný „organizační operační systém“, který:

sjednotí jazyk řízení (co je cíl, co je výkon, co je dohoda, co je rozhodnutí),

zmenší chaos (méně ad-hoc výkřiků, více rytmu),

zvýší pravdivost řízení (reporting z dat, ne z dojmů),

podpoří férovost (odměny navázané na transparentní pravidla),

umožní kaskádu od strategie po denní práci.

1.2 Rozsah

Eviduje a řídí: cíle, metriky, reporting, rituály vedení, překážky, rozhodnutí, rozvoj a odměny.

Integruje data z interních systémů (HRIS, IAM, tiketovací, projektové, CRM, monitoring, finance…).

Poskytuje role-based pohledy pro CEO, manažery i jednotlivce.

1.3 Co systém není

Není to nový Jira/ServiceNow/CRM.

Není to nástroj na mikromanagement „po pěti minutách“.

Není to veřejná „tabule hanby“ – musí podporovat kulturu důvěry.

2) Stakeholdeři a role
2.1 Role v systému

CEO / Board – strategická kaskáda, zdraví organizace, rizika, trendy.

Executive / Director / Head – cíle domén, kapacity, překážky, eskalace.

Manager / Team Lead – vedení lidí (rituály), výkon týmu, dohody, odstranění blokérů.

Individual Contributor – vlastní cíle, progres, překážky, dohody z 1:1, rozvoj.

HR / People Ops – rámce hodnocení, kompetence, cykly, odměny (role-based).

Finance – pravidla bonusů, rozpočty, schvalování (omezený přístup).

Admin (System + Org) – konfigurace, konektory, oprávnění, audit.

Auditor / Compliance – read-only auditní přístup (striktně omezeno).

Integrator (API / ETL) – správa konektorů, mapování, datové toky.

2.2 Princip oprávnění (základ)

Uživatel vidí:

vlastní data,

data svého týmu (pokud je manažer),

agregace vyšších úrovní (pokud je management),

citlivá HR/odměnová data pouze se zvláštním oprávněním.

Každá entita nese klasifikaci citlivosti (Public/Internal/Confidential/HR-Sensitive).

FR-081 Hodnocení rozdělit na: výsledky (cíle) + chování/kompetence + rozvoj.
FR-082 „Bez překvapení“: roční shrnutí se skládá z průběžných záznamů.

4) Integrace a API napojení (co nejpraktičtěji)
4.1 Integrační architektura – doporučený vzor

Aby systém nezkolaboval pod různorodostí API:

Connector framework (pluginy) + jednotná smlouva „Connector API“.

Ingestion pipeline:

Pull (scheduled) + Push (webhook) + Event stream (volitelně).

Canonical data model uvnitř aplikace, mapování z externích systémů.

Idempotence (každý import opakovatelný bez duplicit).

Data provenance: u každé hodnoty uložit „odkud je“ + timestamp.

4.2 Typy konektorů

IAM/Identity: Azure AD / AD / Okta / LDAP (uživatelé, skupiny, role).

HRIS: Workday / SAP SuccessFactors / BambooHR… (org struktura, pracovní poměr).

Projektové/Dev: Jira, GitLab/GitHub, Azure DevOps (epicy, issues, PR, lead time).

ITSM: ServiceNow, Jira Service Management (incidenty, změny, requesty).

Monitoring/SRE: Prometheus, Grafana, Datadog, New Relic (SLO, availability, MTTR).

CRM: Salesforce, HubSpot (pipeline, churn, revenue).

Finance: SAP, Oracle, interní BI (budget vs actual).

Komunikace: Slack, MS Teams, Email, kalendáře (O365/Google).

4.3 Požadavky na integrační vrstvu (detail)

INT-001 Konektor musí podporovat: autentizaci, test spojení, mapování, plán běhu.
INT-002 Podporované auth: OAuth2 (client credentials / auth code), API token, mTLS, Basic (jen výjimečně).
INT-003 Secret management (Vault/KMS), nikdy neukládat tajemství plaintext.
INT-004 Webhook endpointy s podpisem (HMAC/JWT), allow-list IP (volitelně).
INT-005 Rate limiting + backoff + retry politika.
INT-006 Schopnost „dry-run importu“ (ukázka mapování bez zápisu).
INT-007 Schema evolution: když se změní pole v externím API, konektor nesmí rozbít core.
INT-008 Monitoring integrací: poslední úspěch, chybovost, latence, počet záznamů.
INT-009 Dead-letter queue pro chybné události + možnost reprocess.

4.4 Jaké metriky se typicky počítají z integrací (příklady)

IT/Ops: availability, SLO, incident count, MTTR, change failure rate.

Dev: deployment frequency, lead time for changes, PR cycle time.

Service desk: time to first response, time to resolution, backlog.

Obchod: pipeline, win rate, churn, ARR/MRR.
Tyto metriky se mapují na cíle a reporting.

4.5 Veřejné API aplikace (pro vaše interní systémy)

Systém musí nabídnout API pro:

API-001 CRUD cíle, metriky, překážky, rozhodnutí, rituály (role-based).

API-002 Ingestion API (příjem událostí z externích systémů).

API-003 Query API (agregace pro BI / dashboardy).

API-004 Webhooky ven (např. „cíl je červený“, „blokér eskaloval“).

API-005 Audit API (pro compliance nástroje, read-only).
Technicky: REST + případně GraphQL pro složitější dotazy; eventy přes Kafka/Rabbit (volitelně).
